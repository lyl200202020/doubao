## 1 系统概述

### 1.1 项目背景

本系统为B端企业级客户运营管理后台，采用微服务架构建设，旨在为金融机构（或企业）提供完整的客户管理、授权管理及系统基础管理能力。

### 1.2 系统架构

| 层级 | 技术栈 |
|------|--------|
| 前端 | Vue3 + Vite + TS + Pinia + VueRouter + Element Plus |
| 后端 | JDK 17 + SpringBoot 3.3 + SpringCloud Alibaba + Dubbo 3.2+ |
| 中间件 | Nacos + Sentinel + Seata |
| 数据库 | MySQL 8.0+ |
| 安全 | Spring Security + OAuth2 + 国密算法 |

### 1.3 系统模块

```
客户运营管理后台系统
├── 系统管理
│   ├── 用户管理
│   ├── 角色管理
│   ├── 菜单管理
│   └── 机构管理
├── 授权管理
│   ├── 待办任务
│   ├── 我的已办
│   └── 授权历史
└── 客户管理
    ├── 客户管理
    └── 客户签约
```

## 2 数据模型设计

### 2.1 机构管理（Organization）

#### 2.1.1 实体定义

机构是组织架构的基本单元，采用树形结构管理。

#### 2.1.2 数据结构

| 字段名 | 字段类型 | 必填 | 说明 |
|--------|----------|------|------|
| org_id | VARCHAR(32) | 是 | 机构ID（UUID） |
| org_code | VARCHAR(32) | 是 | 机构代码（唯一） |
| org_name | VARCHAR(128) | 是 | 机构名称 |
| org_level | TINYINT | 是 | 机构级别：0-总行，1-分行，2-支行，3-分理处 |
| parent_org_id | VARCHAR(32) | 否 | 上级机构ID（顶级为NULL） |
| org_status | TINYINT | 是 | 状态：0-停用，1-启用 |
| sort_order | INT | 是 | 排序号 |
| create_time | DATETIME | 是 | 创建时间 |
| update_time | DATETIME | 是 | 更新时间 |

#### 2.1.3 业务规则

1. **机构级别**：固定4级（0-总行、1-分行、2-支行、3-分理处），不可扩展
2. **层级约束**：新增机构时，其级别 = 上级机构级别 + 1
3. **上级约束**：只能向所属机构的上级机构发起授权
4. **数据完整性**：删除机构时检查是否存在下级机构或关联用户

---

### 2.2 用户管理（User）

#### 2.2.1 实体定义

用户是系统操作的主体，每个用户必须属于一个机构和一个角色。

#### 2.2.2 数据结构

| 字段名 | 字段类型 | 必填 | 说明 |
|--------|----------|------|------|
| user_id | VARCHAR(32) | 是 | 用户ID（UUID） |
| user_code | VARCHAR(32) | 是 | 用户工号（唯一） |
| user_name | VARCHAR(64) | 是 | 用户姓名 |
| password | VARCHAR(128) | 是 | 密码（国密SM3加密） |
| mobile | VARCHAR(20) | 否 | 手机号 |
| email | VARCHAR(128) | 否 | 邮箱 |
| org_id | VARCHAR(32) | 是 | 所属机构ID |
| role_id | VARCHAR(32) | 是 | 所属角色ID |
| auth_level | TINYINT | 是 | 授权级别（1-16级），值越大级别越高 |
| user_status | TINYINT | 是 | 状态：0-停用，1-启用 |
| last_login_time | DATETIME | 否 | 最后登录时间 |
| create_time | DATETIME | 是 | 创建时间 |
| update_time | DATETIME | 是 | 更新时间 |

#### 2.2.3 业务规则

1. **唯一性**：用户工号（user_code）在全系统唯一
2. **机构归属**：用户只能属于一个机构，机构删除时需先转移或删除用户
3. **角色归属**：用户只能绑定一个角色，角色删除时需先解绑用户
4. **授权级别**：1-16级，用于授权审批时判断是否满足授权条件
5. **密码策略**：首次登录强制修改密码，密码复杂度要求（长度≥8，包含数字+字母+特殊字符）
6. **登录限制**：同一账号最多5个并发会话

---

### 2.3 角色管理（Role）

#### 2.3.1 实体定义

角色用于控制用户可见的功能菜单，与授权体系无关。

#### 2.3.2 数据结构

| 字段名 | 字段类型 | 必填 | 说明 |
|--------|----------|------|------|
| role_id | VARCHAR(32) | 是 | 角色ID（UUID） |
| role_code | VARCHAR(32) | 是 | 角色代码（唯一） |
| role_name | VARCHAR(64) | 是 | 角色名称 |
| role_desc | VARCHAR(256) | 否 | 角色描述 |
| role_status | TINYINT | 是 | 状态：0-停用，1-启用 |
| create_time | DATETIME | 是 | 创建时间 |
| update_time | DATETIME | 是 | 更新时间 |

#### 2.3.3 业务规则

1. **功能可见性**：角色决定用户能看到哪些菜单（可见/不可见）
2. **与授权无关【重要】**：
   - 角色只控制"**看得见**"还是"**看不见**"
   - 角色**不参与**授权审批流程
   - 授权审批由**用户的授权级别**决定，与角色无关
   - 例：即使没有"授权管理"菜单权限的用户，只要有授权级别，也能给其他人授权（需通过其他方式触发）
3. **超级管理员**：系统内置超级管理员角色，拥有所有菜单权限
4. **删除限制**：删除角色前需检查是否有用户绑定

---

### 2.4 菜单管理（Menu）

#### 2.4.1 实体定义

菜单是系统功能的入口，分为目录、菜单、按钮三种类型。

#### 2.4.2 数据结构

| 字段名 | 字段类型 | 必填 | 说明 |
|--------|----------|------|------|
| menu_id | VARCHAR(32) | 是 | 菜单ID（UUID） |
| menu_name | VARCHAR(64) | 是 | 菜单名称 |
| menu_type | TINYINT | 是 | 类型：1-目录，2-菜单，3-按钮 |
| parent_menu_id | VARCHAR(32) | 否 | 父菜单ID（顶级为NULL） |
| menu_url | VARCHAR(256) | 否 | 菜单URL/路由地址 |
| menu_icon | VARCHAR(64) | 否 | 菜单图标 |
| sort_order | INT | 是 | 排序号 |
| visible | TINYINT | 是 | 是否可见：0-隐藏，1-显示 |
| perms | VARCHAR(128) | 否 | 权限标识（按钮用） |
| create_time | DATETIME | 是 | 创建时间 |
| update_time | DATETIME | 是 | 更新时间 |

#### 2.4.3 业务规则

1. **菜单类型**：
   - **目录（type=1）**：用于分组，无实际功能，仅作为父节点
   - **菜单（type=2）**：左侧导航菜单，对应具体页面
   - **按钮（type=3）**：页面内操作按钮，如"新增"、"修改"、"删除"
2. **权限标识**：按钮级别菜单需配置perms，用于前端权限控制
3. **角色关联**：菜单可以绑定多个角色，决定哪些角色可见该菜单
4. **层级限制**：菜单层级最多4级

#### 2.4.4 菜单-角色关联表（sys_menu_role）

| 字段名 | 字段类型 | 必填 | 说明 |
|--------|----------|------|------|
| menu_id | VARCHAR(32) | 是 | 菜单ID |
| role_id | VARCHAR(32) | 是 | 角色ID |

## 3 权限体系设计

### 3.1 授权管理概述

本系统授权体系采用**分级授权**机制，支持同步授权和异步授权两种模式，适用于交易授权和金额授权场景。

### 3.2 授权相关实体

#### 3.2.1 授权模板（AuthTemplate）

| 字段名 | 字段类型 | 必填 | 说明 |
|--------|----------|------|------|
| template_id | VARCHAR(32) | 是 | 模板ID |
| template_code | VARCHAR(32) | 是 | 模板代码 |
| template_name | VARCHAR(128) | 是 | 模板名称 |
| template_type | TINYINT | 是 | 模板类型：1-交易授权，2-金额授权 |
| auth_type | TINYINT | 是 | 授权方式：1-同步授权，2-异步授权 |
| template_status | TINYINT | 是 | 状态：0-停用，1-启用 |
| create_time | DATETIME | 是 | 创建时间 |
| update_time | DATETIME | 是 | 更新时间 |

#### 3.2.2 授权级别配置（AuthLevelConfig）

| 字段名 | 字段类型 | 必填 | 说明 |
|--------|----------|------|------|
| config_id | VARCHAR(32) | 是 | 配置ID |
| template_id | VARCHAR(32) | 是 | 所属模板ID |
| level_no | TINYINT | 是 | 授权级别序号（1-N），1表示第一级，2表示第二级... |
| auth_level | TINYINT | 是 | 需要的授权级别（1-16），只有用户授权级别=此值才能审批 |
| min_amount | DECIMAL(18,2) | 否 | 金额区间起始值（金额授权用） |
| max_amount | DECIMAL(18,2) | 否 | 金额区间结束值（金额授权用） |
| create_time | DATETIME | 是 | 创建时间 |

**多级授权配置说明**：

| 场景 | level_no | auth_level | 说明 |
|------|----------|------------|------|
| 交易需要2,3,5三级授权 | 1 | 2 | 第1级：只有授权级别=2的用户可审批 |
| | 2 | 3 | 第2级：只有授权级别=3的用户可审批 |
| | 3 | 5 | 第3级：只有授权级别=5的用户可审批 |

**审批流程（逐级审批）**：
1. 第1级授权：所有=auth_level的用户可见待办，**谁先批完算谁通过**
2. 第1级通过后 → 进入第2级授权
3. 第2级通过后 → 进入第3级授权
4. 全部通过 → 交易完成
5. 任意一级拒绝 → 交易回滚

**关键规则**：
- 同一级别的多个授权人：**谁先批完，另外的人就看不到了**（任务被锁定）
- **严格匹配**：只有授权级别=该级别人才能审批（2级任务只有2级用户能审批）
- 级别序号（level_no）必须连续，从1开始递增

#### 3.2.3 菜单-授权模板关联表（sys_menu_auth_template）

| 字段名 | 字段类型 | 必填 | 说明 |
|--------|----------|------|------|
| menu_id | VARCHAR(32) | 是 | 菜单ID（对应功能点） |
| template_id | VARCHAR(32) | 是 | 授权模板ID |

**关联说明**：
- 一个授权模板可以关联多个菜单（功能点）
- 一个菜单只能关联一个授权模板
- 菜单类型为"按钮"时需配置授权模板

#### 3.2.4 授权任务（AuthTask）

| 字段名 | 字段类型 | 必填 | 说明 |
|--------|----------|------|------|
| task_id | VARCHAR(32) | 是 | 任务ID |
| task_no | VARCHAR(32) | 是 | 任务流水号 |
| template_id | VARCHAR(32) | 是 | 授权模板ID |
| biz_type | VARCHAR(32) | 是 | 业务类型（如：客户签约） |
| biz_id | VARCHAR(32) | 是 | 业务数据ID |
| biz_data | JSON | 是 | 业务数据快照（表单数据） |
| biz_screenshots | JSON | 否 | 业务页面截图数组（[{"page":1,"img":"Base64..."},...]），支持多图 |
| apply_user_id | VARCHAR(32) | 是 | 申请人ID |
| apply_org_id | VARCHAR(32) | 是 | 申请人所属机构ID |
| current_level | TINYINT | 是 | 当前待授权级别 |
| task_status | TINYINT | 是 | 状态：1-待授权，2-授权通过，3-授权拒绝，4-已取消 |
| apply_time | DATETIME | 是 | 申请时间 |
| auth_time | DATETIME | 否 | 授权时间 |
| auth_user_id | VARCHAR(32) | 否 | 授权人ID |
| auth_opinion | VARCHAR(256) | 否 | 授权意见 |

#### 3.2.5 授权历史（AuthHistory）

| 字段名 | 字段类型 | 必填 | 说明 |
|--------|----------|------|------|

| task_id | VARCHAR(32) | 是 | 授权任务ID |
| auth_level | TINYINT | 是 | 授权级别 |
| auth_user_id | VARCHAR(32) | 是 | 授权人ID |
| auth_result | TINYINT | 是 | 授权结果：1-通过，2-拒绝 |
| auth_opinion | VARCHAR(256) | 否 | 授权意见 |
| auth_time | DATETIME | 是 | 授权时间 |

### 3.3 菜单授权触发机制

#### 3.3.1 授权触发流程

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  用户点击   │────▶│  系统查询   │────▶│  判断是否   │────▶│  触发授权   │
│  功能按钮   │     │  菜单关联的 │     │  需要授权   │     │  流程       │
│             │     │  授权模板   │     │             │     │             │
└─────────────┘     └─────────────┘     └─────────────┘     └─────────────┘
```

**触发逻辑**：
1. 用户点击功能按钮（如"签约确定"）
2. 系统根据菜单ID查询关联的授权模板（sys_menu_auth_template）
3. 如无关联模板 → 无需授权，直接执行
4. 如有关联模板 → 根据模板类型和参数判断需要的授权级别
5. 根据授权方式（同步/异步）执行对应流程

#### 3.3.2 授权判断示例

**示例1：交易授权（同步）**
- 菜单"客户签约-确定" 关联授权模板T001（交易授权，同步）
- 模板配置：1级授权（auth_level≥5）
- 用户A（授权级别3）点击"确定" → 弹出授权框，需授权级别≥5的用户授权

**示例2：金额授权（异步）**
- 菜单"客户签约-确定" 关联授权模板T002（金额授权，异步）
- 模板配置：
  - 金额区间[0, 10000)：无需授权
  - 金额区间[10000, 50000)：1级授权（auth_level≥3）
  - 金额区间[50000, ∞)：2级授权（auth_level≥8）
- 用户提交签约金额80000 → 进入2级授权流程，生成待办任务

#### 3.3.3 同步授权流程

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  经办人员   │     │   系统      │     │  授权人员   │
│  发起交易   │────▶│  弹出授权   │────▶│  输入柜员   │
│             │     │  对话框     │     │  号和密码   │
└─────────────┘     └─────────────┘     └─────────────┘
                                               │
                    ┌─────────────┐            │
                    │  授权通过   │◀───────────┘
                    │  交易完成   │
                    └─────────────┘
```

**流程说明**：
1. 经办人员发起需要授权的交易
2. 系统根据授权模板判断需要授权，弹出授权对话框
3. 授权人员输入柜员号和密码进行授权
4. 系统校验授权人权限和授权级别
5. 授权通过则交易完成，授权拒绝则交易回滚

#### 3.3.4 异步授权流程

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  经办人员   │     │   系统      │     │  待办任务   │
│  提交交易   │────▶│  生成待办   │────▶│  列表展示   │
│  +截图      │     │  +截图      │     │             │
└─────────────┘     └─────────────┘     └─────────────┘
                                               │
┌─────────────┐     ┌─────────────┐            │
│  交易完成   │◀────│  授权通过   │◀───────────┘
│             │     │  或拒绝     │
└─────────────┘     └─────────────┘
```

**流程说明**：
1. 经办人员提交交易，**前端对当前页面进行截图**（Base64）
2. 系统生成授权任务（包含表单数据+截图），交易状态变为"待授权"
3. 授权人员从待办任务进入，**查看截图直观了解经办人操作内容**
4. 授权人员选择通过或拒绝，并填写意见
5. 根据授权结果更新业务数据状态

**多图截图机制**：
- 适用场景：多步骤/多页面录入的业务（如向导式表单）
- 截图时机：经办人员点击"提交"时，前端依次对每个页面进行截图
- 截图存储：JSON数组格式 `[{page:1, img:"Base64..."}, {page:2, img:"Base64..."}, ...]`
- 展示方式：
  - 授权审批页面显示多页签（页签1、页签2...）
  - 每个页签显示对应页面的截图缩略图
  - 点击可放大查看原图
- 技术实现：前端使用html2canvas依次截图，组装为JSON数组

### 3.4 授权规则

#### 3.4.1 交易授权

- 配置交易授权模板，定义需要多级（N级）授权
- 每级配置需要的授权级别（auth_level）
- 例：1级授权需要用户授权级别≥5，2级授权需要用户授权级别≥10

#### 3.4.2 金额授权

- 配置金额授权模板，定义金额区间与授权级别映射
- 例：
  - 金额区间 [0, 10000)：1级授权（授权级别≥3）
  - 金额区间 [10000, 50000)：2级授权（授权级别≥5）
  - 金额区间 [50000, 100000)：3级授权（授权级别≥8）
  - 金额区间 [100000, ∞)：4级授权（授权级别≥12）

#### 3.4.3 授权机构限制

- 用户只能向**所属机构的上级机构**发起授权申请
- 授权人员只能处理其所属机构及下级机构的授权任务
- 例：支行用户只能向分行发起授权，分行只能向总行发起授权

## 4 功能模块设计

### 4.1 系统管理

#### 4.1.1 用户管理

**页面路径**：系统管理 > 用户管理

**功能清单**：

| 功能 | 说明 |
|------|------|
| 用户列表 | 分页展示用户，支持按机构、角色、状态筛选 |
| 新增用户 | 填写用户基本信息，绑定机构和角色 |
| 修改用户 | 编辑用户信息，可修改机构和角色 |
| 删除用户 | 逻辑删除用户 |
| 重置密码 | 重置用户密码为默认密码 |
