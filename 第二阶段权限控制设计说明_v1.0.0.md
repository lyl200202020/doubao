## 一、核心概念区分

### 1.1 权限控制 vs 授权管理

**这是两个完全不同的概念，必须明确区分：**

| 维度 | **权限控制（第二阶段）** | **授权管理（第三阶段）** |
|------|--------------------------|--------------------------|
| **定义** | 控制用户能看到什么菜单、能访问什么页面、能点击什么按钮 | 控制业务审批流程，基于授权级别的审批权限 |
| **核心问题** | "你能看到什么功能？" | "你能审批什么业务？" |
| **实现方式** | RBAC 模型（用户→角色→菜单） | 授权级别模型（用户授权级别 1-16 级） |
| **应用场景** | 系统功能访问控制 | 业务审批流程控制 |
| **技术实现** | 菜单可见性、路由守卫、按钮指令、接口拦截 | 授权级别比对、审批流程引擎 |
| **数据载体** | 角色、菜单、权限标识 | 授权级别、待授权业务、授权记录 |
| **用户感知** | 登录后看到的菜单不同 | 提交业务后等待审批 |

### 1.2 形象比喻

| 场景 | 权限控制 | 授权管理 |
|------|----------|----------|
| **银行大厅** | 你是普通客户只能在大厅办理业务，VIP 客户可以进入 VIP 室 | 你提交贷款申请后，需要不同级别的领导审批 |
| **公司大楼** | 普通员工只能进入办公区，高管可以进入 executive lounge | 你提交的报销单需要部门经理审批 |
| **网站访问** | 普通用户看到基础功能，会员用户看到高级功能 | 你发布的文章需要审核后才能上线 |

## 二、权限控制模型（第二阶段）

### 2.1 RBAC 模型

```
┌─────────────┐
│   用户      │
│  (User)     │
└──────┬──────┘
       │ 1
       │
       │ n
┌──────┴──────┐
│   角色      │
│  (Role)     │
└──────┬──────┘
       │ n
       │
       │ n
┌──────┴──────┐
│   菜单      │
│  (Menu)     │
└─────────────┘
```

### 2.2 权限控制层级

```
第一层：菜单可见性控制
  ↓ 用户登录后只能看到其角色关联的菜单
  ↓
第二层：路由访问控制
  ↓ 用户只能访问其角色允许的页面路由
  ↓ 访问未授权路由时重定向到 403 页面
  ↓
第三层：按钮权限控制
  ↓ 用户只能看到和点击其角色允许的按钮
  ↓ 无权限按钮自动隐藏或禁用
  ↓
第四层：接口访问控制
  ↓ 后端校验用户是否有权限调用接口
  ↓ 无权限调用返回 403 状态码
```

### 2.3 权限标识（perms）

**格式**：`模块：操作：资源`

**示例**：
- `sys:org:view` - 查看机构
- `sys:org:add` - 新增机构
- `sys:org:edit` - 编辑机构
- `sys:org:delete` - 删除机构
- `sys:user:view` - 查看用户
- `sys:user:add` - 新增用户
- `sys:user:edit` - 编辑用户
- `sys:user:delete` - 删除用户
- `sys:menu:view` - 查看菜单
- `sys:menu:add` - 新增菜单
- `sys:menu:edit` - 编辑菜单
- `sys:menu:delete` - 删除菜单
- `sys:role:view` - 查看角色
- `sys:role:add` - 新增角色
- `sys:role:edit` - 编辑角色
- `sys:role:delete` - 删除角色

## 三、权限控制实现方案

### 3.1 数据库设计

#### 3.1.1 菜单表（sys_menu）

```sql
CREATE TABLE sys_menu (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '主键 ID',
    menu_name VARCHAR(50) NOT NULL COMMENT '菜单名称',
    parent_id BIGINT NOT NULL DEFAULT 0 COMMENT '父菜单 ID',
    menu_type VARCHAR(20) NOT NULL COMMENT '菜单类型：1-目录，2-菜单，3-按钮',
    perms VARCHAR(100) COMMENT '权限标识（sys:org:view）',
    path VARCHAR(200) COMMENT '路由路径',
    component VARCHAR(200) COMMENT '组件路径',
    icon VARCHAR(50) COMMENT '菜单图标',
    sort_order INT NOT NULL DEFAULT 0 COMMENT '排序',
    visible VARCHAR(20) NOT NULL DEFAULT '1' COMMENT '是否可见：0-隐藏，1-显示',
    created_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    updated_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    deleted TINYINT NOT NULL DEFAULT 0 COMMENT '删除标记'
) COMMENT '菜单表';
```

**关键字段**：
- `perms`：权限标识，用于按钮级别权限控制
- `menu_type`：菜单类型，区分目录、菜单、按钮
- `path`：路由路径，用于动态路由生成
- `component`：组件路径，用于动态加载组件

#### 3.1.2 角色表（sys_role）

```sql
CREATE TABLE sys_role (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '主键 ID',
    role_code VARCHAR(50) NOT NULL COMMENT '角色代码',
    role_name VARCHAR(50) NOT NULL COMMENT '角色名称',
    role_status VARCHAR(20) NOT NULL DEFAULT '1' COMMENT '角色状态：0-停用，1-启用',
    created_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    updated_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    deleted TINYINT NOT NULL DEFAULT 0 COMMENT '删除标记'
) COMMENT '角色表';
```

#### 3.1.3 用户 - 角色关联表（sys_user_role）

#### 3.2.2 权限注解开发

```java
@Target({ElementType.METHOD})
@Retention(RetentionPolicy.RUNTIME)
@Documented
public @interface RequirePermission {
    /**
     * 权限标识
     */
    String value();
    
    /**
     * 逻辑关系（AND 或 OR）
     */
    Logical logical() default Logical.OR;
}
```

**使用示例**：
```java
@RestController
@RequestMapping("/api/sys/org")
public class OrgController {
    
    @RequirePermission("sys:org:view")
    @GetMapping("/tree")
    public Result<List<OrgVO>> getOrgTree() {
        // ...
    }
    
    @RequirePermission("sys:org:add")
    @PostMapping
    public Result<Void> addOrg(@RequestBody OrgDTO dto) {
        // ...
    }
    
    @RequirePermission("sys:org:delete")
    @DeleteMapping("/{id}")
    public Result<Void> deleteOrg(@PathVariable Long id) {
        // ...
    }
}
```

#### 3.2.3 权限拦截器

```java
@Component
public class PermissionInterceptor implements HandlerInterceptor {
    
    @Override
    public boolean preHandle(HttpServletRequest request, 
                           HttpServletResponse response, 
                           Object handler) throws Exception {
        // 获取当前用户权限
        Set<String> userPerms = getCurrentUserPermissions();
        
        // 获取接口需要的权限
        RequirePermission requirePermission = getRequirePermission(handler);
        
        // 校验权限
        if (requirePermission != null && !hasPermission(userPerms, requirePermission)) {
            response.setStatus(HttpStatus.FORBIDDEN.value());
            response.setContentType("application/json;charset=UTF-8");
            response.getWriter().write("{\"code\":403,\"msg\":\"无权限\"}");
            return false;
        }
        
        return true;
    }
}
```

### 3.3 前端实现

#### 3.3.1 路由权限守卫

```typescript
// router/permission.ts
import router from './index'
import { useUserStore } from '@/store/modules/user'

const whiteList = ['/login', '/404'] // 白名单路由

router.beforeEach(async (to, from, next) => {
  const userStore = useUserStore()
  const hasToken = userStore.token
  
  
  if (hasToken) {
    if (to.path === '/login') {
      next({ path: '/' })
    } else {
      const hasRoles = userStore.roles && userStore.roles.length > 0
      
      if (hasRoles) {
        next()
      } else {
        try {
          // 获取用户信息和菜单
          const { menus, perms } = await userStore.getInfo()
          
          // 动态生成路由
          const accessRoutes = generateRoutes(menus)
          accessRoutes.forEach(route => {
            router.addRoute(route)
          })
          
          next({ ...to, replace: true })
        } catch (error) {
          // 获取失败，清空 token 并跳转登录页
          await userStore.resetToken()
          next(`/login?redirect=${to.path}`)
        }
      }
    }
  } else {
    // 没有 token
    if (whiteList.indexOf(to.path) !== -1) {
      next()
    } else {
      next(`/login?redirect=${to.path}`)
    }
  }
})
```

#### 3.3.2 动态菜单生成

```typescript
// store/modules/app.ts
import { defineStore } from 'pinia'
import { getMenus } from '@/api/sys/menu'

export const useAppStore = defineStore('app', {
  state: () => ({
    menus: [] as Menu[],
    permissions: [] as string[]
  }),
  
  actions: {
    async generateMenus() {
      const res = await getMenus()
      this.menus = res.data
      
      // 提取所有权限标识
      this.permissions = this.extractPermissions(this.menus)
      
      return this.menus
    },
    
    extractPermissions(menus: Menu[]): string[] {
      const perms: string[] = []
      menus.forEach(menu => {
        if (menu.perms) {
          perms.push(menu.perms)
        }
        if (menu.children && menu.children.length > 0) {
          perms.push(...this.extractPermissions(menu.children))
        }
      })
      return perms
    }
  }
})
```

#### 3.3.3 按钮权限指令

```typescript
// directives/permission.ts
import { useUserStore } from '@/store/modules/user'
import { Directive, DirectiveBinding } from 'vue'

export const permission: Directive = {
  mounted(el: HTMLElement, binding: DirectiveBinding) {
    const { value } = binding
    const userStore = useUserStore()
    const permissions = userStore.permissions
    
    if (value && value instanceof Array && value.length > 0) {
      const permissionFlag = value
      
      const hasPerm = permissions.some(perm => {
        return permissionFlag.includes(perm)
      })
      
      if (!hasPerm) {
        el.parentNode && el.parentNode.removeChild(el)
      }
    } else {
      throw new Error('请设置权限标签值')
    }
  }
}
```

**使用示例**：
```vue
<template>
  <div>
    <!-- 新增机构按钮 -->
    <el-button 
      v-permission="['sys:org:add']"
      type="primary" 
      @click="handleAdd"
    >
      新增机构
    </el-button>
    
    <!-- 编辑机构按钮 -->
    <el-button 
      v-permission="['sys:org:edit']"
      type="primary" 
      @click="handleEdit"
    >
      编辑机构
```

## 五、授权管理预告（第三阶段）

### 5.1 授权管理模型

```
用户（授权级别 1-16 级）
  ↓
提交业务（如：贷款申请、费用报销）
  ↓
授权流程（根据业务金额/重要性确定需要的授权级别）
  ↓
授权人（授权级别 >= 业务需要的授权级别）审批
  ↓
业务生效
```

### 5.2 授权管理与权限控制的区别

| 场景 | 权限控制（第二阶段） | 授权管理（第三阶段） |
|------|--------------------------|--------------------------|
| **用户登录** | 看到不同的菜单 | 无影响 |
| **页面访问** | 只能访问有权限的页面 | 无影响 |
| **按钮显示** | 只显示有权限的按钮 | 无影响 |
| **业务提交** | 无影响 | 提交后进入授权流程 |
| **业务审批** | 无影响 | 根据授权级别审批 |

**示例**：
- **权限控制**：柜员只能看到"业务办理"菜单，行长能看到"业务办理"+"业务审批"菜单
- **授权管理**：柜员提交一笔 100 万贷款申请，需要授权级别≥10 级的行长审批

## 六、总结

### 6.1 第二阶段权限控制核心要点

1. **RBAC 模型**：用户→角色→菜单
2. **四级权限控制**：菜单可见性、路由访问、按钮权限、接口校验
3. **权限标识**：`模块：操作：资源` 格式
4. **技术实现**：
   - 后端：权限注解 + 拦截器
   - 前端：路由守卫 + 按钮指令
5. **验收标准**：不同角色用户看到不同菜单，无权限按钮隐藏，无权限接口返回 403

### 6.2 第三阶段授权管理预告

1. **授权级别模型**：用户授权级别 1-16 级
2. **授权流程**：业务提交→授权审批→业务生效
3. **授权规则**：根据业务金额/重要性确定需要的授权级别
4. **授权记录**：记录完整的授权审批历史
